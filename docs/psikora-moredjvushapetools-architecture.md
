# 2. High-Level Architecture

The project is a Python-based GUI toolkit operating on
a MySQL/MariaDB database created by `exportshapes`.

System layers:

    MySQL database
            ↓
    internal/data.py
    internal/database_manipulation.py
            ↓
    GUI components (browser/, labeller/, djvusmooth/)
            ↓
    Rendering logic (shape panels, previews)

The architecture is tightly coupled:
- GUI logic directly invokes database queries
- Rendering often depends directly on database results

There is no strong separation between:
- Model
- Data access
- Rendering
- View logic

---

# 3. Database Model Recap

Database created by exportshapes:

    documents
        ↓
    dictionaries
        ↓
    shapes
        ↓
    blits

Important:

- shapes.bits is a LONGBLOB containing JB2 bitmap data.
- blits link shapes to page_number.
- pages table is only used for inherited dictionaries.

Rendering depends primarily on:
- shapes
- blits
- dictionaries

# 4. Shape Rendering Pipeline (Logical Reconstruction)

When a shape must be displayed:

1. GUI requests shape ID.
2. SQL query fetches:
       bits, width, height
3. Binary JB2 blob is returned.
4. image_conversion.py decodes blob.
5. wx.Bitmap / wx.Image object created.
6. Paint handler draws bitmap.

Likely draw call:

    dc.DrawBitmap(bitmap, x, y, True)

If bitmap decoding happens during repaint,
performance degrades dramatically.

# 6. Probable Rendering Workflow

User scrolls
    ↓
Visible shape IDs determined
    ↓
Database queried
    ↓
Blob decoded
    ↓
Bitmap created
    ↓
Paint event draws bitmap

If this repeats per repaint:
→ severe slowdown

# 7. Major Performance Risk Points

## 7.1 Blob Decoding Per Paint

If bits → bitmap conversion happens every time,
CPU usage explodes.

Correct design:
Cache decoded bitmaps.

---

## 7.2 No Bitmap Cache

Ideal structure:

    shape_bitmap_cache = {shape_id: wx.Bitmap}

Without this:
Every scroll = full recomputation.

---

## 7.3 Row-by-Row SQL

Anti-pattern:

    for shape_id in shape_ids:
        SELECT ... WHERE id = ?

Correct pattern:

    SELECT ... WHERE id IN (...)

---

## 7.4 Full Repaint Strategy

If the entire panel is redrawn,
even off-screen elements are rendered.

Better:
Virtual scrolling / draw only visible shapes.

---

## 7.5 Synchronous DB in GUI Thread

If SQL queries block the UI:
Application appears frozen.

# 8. Likely GUI Rendering Model

Shape panel:

- Calculates grid layout
- Determines shape IDs
- Requests each shape
- Converts and draws

Labeller preview:

- Combines multiple shapes
- Possibly rescales
- Renders composite image

Composite rendering without caching
is expensive.

---

# 9. Structural Weaknesses

- Tight coupling between DB and GUI
- Likely no caching layer
- No clear MVC separation
- Possible N² patterns in shape comparisons
- No asynchronous loading
- Python 2 era design assumptions

---

# 10. Why It May Be Prohibitively Slow

If:
- Hundreds of shapes are displayed
- Each repaint decodes blobs
- Each shape triggers a separate SQL query
- Entire panel is redrawn frequently

Then complexity becomes:

    O(number_of_shapes × repaint_events)

This becomes catastrophic at scale.

---

# 11. Recommended Modern Rewrite Strategy

## 11.1 Separate Layers

- DB access layer
- Shape decoding layer
- Rendering layer
- GUI layer

## 11.2 Introduce Caching

- Bitmap cache
- Dictionary-level cache
- Page-level composite cache

## 11.3 Bulk SQL

Avoid per-shape queries.

## 11.4 Lazy Rendering

Render only shapes visible in viewport.

## 11.5 Background Loading

Use worker thread for DB + decoding.
